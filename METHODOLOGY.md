# Claude Code 项目开发治理体系方法论

> **这是一份可分享的通用方法论**，不含任何具体项目的业务细节。
> 适用对象：用 Claude Code（或同类 AI 编码 agent）做**多会话、长周期、可能多人/多分支**软件开发的团队。
> 它解决一个具体问题：**AI 驱动的开发会随时间腐化，且腐化是结构性的、不可避免的——除非有治理。**
>
> 本文是从一个真实项目（十几个版本迭代、数十个 session、烧掉 ~20 亿 token）的踩坑中提炼的逻辑。
> 你拿去按自己项目特点实例化即可。**抽象原则在前，落地配方在后。**

---

## 0. 一句话

把"事实"和"规则"分开存；规则约束 AI 怎么工作，事实只从代码再生；每个 session 开工前先验证现状再动手；不凭印象、不绕行、不越界、不让主干腐化。

---

## 元立场:三层结构 —— 规范 · 机制 · 审计

ACGM 是**规范层方法论**(normative)。它定义 agent 行为**应当如何**,不规定由谁来发现违规。

在它之下有三层独立的执行手段,互相不替代而是互补:

| 层 | 承担什么 | 例 |
|---|---|---|
| **规范层(ACGM 本体)** | 声明 agent 行为的应然 | "破坏性操作前必先报告真实状态" |
| **机制层**(hooks / drift-check) | 可机械落实的子集 | PreToolUse 强制结构化提问;drift-check 静态扫 |
| **审计层**(人 / agent 自检 / 外部 agent / 未来 agent 回看记录) | 关闭机制无法覆盖的缝隙 | 用户在长跑任务挂掉数小时后主动指出 |

ACGM **不假设审计由谁完成** —— 它只承诺让违规**可被显化、可被留痕、可被追溯**。机制是部分落地,审计是缝隙补全;两者皆为执行手段,不动摇规范本身。

> **审计可靠性梯度(经验):** 人 / 外部 agent / 未来 agent 显著比**当事 agent 自检**可靠。设计上不应把当事 agent 自检作为主要审计源 —— 这是反复观察到的:当事 agent 对自身漂移的识别能力,系统性低于外部视角。

这一立场决定了 ACGM 的发展方向:不为补偿"人不在场"而设计,也不假定"人总在场";只关心**减少 agent 自身机制性漂移**,并让任何审计者(包括人)更有效。

## 1. 为什么 AI 多会话开发会腐化（先认清敌人）

单个 session 上下文有限。项目一长，你会不停开新 session、写交接文档、让 AI 总结进度。腐化从这里开始，有五个机制：

1. **上下文丢失**：新 session 不知道前面发生过什么，靠交接文档和"记忆"重建——重建必然失真。
2. **凭印象编造**：AI 整理交接/文档时，倾向用对话上下文里的残留信息总结，而不是去读代码真值。上下文里恰好留着早期已废弃的说法，就被当成"现状"写进文档。
3. **追加式更新**：AI 改文档倾向加段落、不删旧的。被推翻的旧方案躺在文档里没有"已废弃"标记，下个 session 读到就走错。
4. **交接污染传递**：A 方向的交接里混着 A 特定的临时绕行，B 方向照单全收，跑偏。
5. **主干腐化**：治理/真值长在某个功能分支上，主干被遗弃在很旧的状态。新 session 落在旧主干 = 读到全错信息。

**关键认知：以上不是"操作失误"，是这套工作流的天然成本。** 每次大跨度迁移（平台切换、技术栈替换、人员交接、版本封存）都是腐化爆发点。治理的目标不是消灭腐化（不可能），而是**让腐化显性化、可拦截、可回滚**。

## 2. 四类漂移（诊断核心，最值得迁移的部分）

把所有腐化归到四类。每类有独立防线。任何 session 触及某类，必须停下走对应流程。

| 漂移类型 | 错在哪 | 典型表现 | 防线 |
|---|---|---|---|
| **① 实施层漂移** | 技术不通就绕路 | 自写 polyfill 模拟原生能力、降级 API、try-catch 静默吞错、改业务迁就 bug | **绕行禁令**：先 root cause；探索性 hack 需人明确授权 + 记录 + 定 deadline |
| **② 认知层漂移** | 写文档凭印象，不验证真值 | "应该是 X""我记得是 X"、从旧文档/交接抄技术结论、不给来源 | **真值优先**：任何技术结论必须 grep/读代码带 `文件:行号`；不能引用就显式标"未验证" |
| **③ 结构放置漂移** | 治理/真值住错地方，主干腐化 | 宪法/规则长在功能分支；主干被遗弃在旧状态；新 session 落旧主干读到全错 | **治理居所**：治理只在主干 author；主干永不准腐化；子目录规则文件只做薄指针 |
| **④ 范围漂移** | 不该进这仓库的内容混进来 | 企业经营/商业战略/与软件开发无关的内容累积在代码仓库 | **范围边界**：明确 IN/OUT；判据"为软件能开发上线运行 = IN，为别的 = OUT" |

这张表是整个方法论的诊断仪表盘。**先学会识别是哪一类，再谈怎么修。**

> 元教训（最强的证明）：这套防漂移体系**在被搭建的过程中，自己就发生了②号漂移**——搭建者从旧交接文档抄了一堆技术结论没去读代码。这恰恰证明：**纪律不是给"别人"的，是给当下每一次写字的你的。** 把你项目里类似的真实事故，永久写进治理文件当警示案例——比抽象规则有用十倍。

## 3. 第一原则：按"生命周期"分层存内容

腐化的根因是**不同生命周期的内容堆在一起互相污染**：产品愿景（几乎不变）、架构决策（变了要替换）、当前状态（一直在变）、历史档案（不可变）混在一个文件里，新 session 没法区分哪些是"宪法"哪些是"快照"。

解法：物理分层。每层有不同的变化频率、写入规则、谁能改。

| 层 | 内容 | 变化频率 | 写入规则 | 谁能改 |
|---|---|---|---|---|
| **宪法** | 不可妥协的原则、红线、角色边界、范围边界 | 几乎不变 | 推翻条款要走修宪流程 | **仅项目所有者（人）** |
| **决策日志（ADR）** | 每个架构/技术/产品决策一份 | append-only | 不改旧的；被推翻的标"已被 X 取代"或"已废除"，**不删除原文** | AI 写，人审 |
| **当前状态快照** | 架构/接口/配置现状 | 跟代码同步 | **由代码扫描重新生成**，不增量改；顶部标"基于 commit X 生成于 Y" | 脚本/AI 再生 |
| **版本归档** | 每次封存时的完整状态 | 封存即定格 | 不可变 | 写完锁死 |
| **跨切面契约** | 多端/多模块必须严格对齐的协议 | 改一处即三方影响 | 改动必须显式声明影响面 + 验证 + 人审批 | 受严格流程约束 |
| **活交接** | 当前还有效的 session 接力 | 失效即归档 | 失效就移历史区 | AI 写 |

两个容易混淆但必须分开的历史区：
- **完工归档**：曾经发生过的事实记录（旧 session 报告等）。是"史实"，只读。
- **被推翻文档**：曾经认为对、现在不对的整篇文档。顶部必须贴"已被 X 取代"横幅。

> 区分它俩很重要：史实可以放心翻；被推翻的文档误读会害死下一个 session。

## 4. 第二原则：项目根规则文件 = 元规则 + 指针 + 行为约束，**绝不放事实**

大多数团队把"项目说明文件"（Claude Code 里是 `CLAUDE.md`）当成事实大全：架构、数据库选型、部署方式、模型版本……全写里面。**这是腐化的头号来源**——能从代码反推的东西，在说明文件里复述一遍，就是制造第二个会漂移的真值源。

正确做法：

- **不变的东西**（产品定位、硬约束、不要做的事）→ 进宪法。
- **会变的事实**（架构、接口、配置、版本）→ **不写答案，写指针**：「X 的真值在 `代码路径`，改动前先读」。
- 根规则文件本身只装三样：
  1. **元规则**：怎么维护这个文件本身（改前先看 git log；禁止 inline 写事实；推翻条款必须删旧文不能追加叠加）
  2. **指针**：去哪查真值（指向代码 / 各层文档）
  3. **行为约束**：每个 session 必须遵守的工作纪律

根规则文件越短越好，越长越容易藏化石。它退化为"协议层"，不承载知识。

## 5. 第三原则：真值优先（绝对版，无灰色地带）

任何写进文档 / commit message / 代码注释 / 给人的报告 里的技术结论：

- ❌ 不允许凭对话上下文记忆
- ❌ 不允许从"历史归档/被推翻文档/旧交接/旧版本快照"抄技术结论
- ❌ 不允许用"应该是""通常是""我记得是""按惯例"这类措辞
- ❌ 不允许写技术结论但不给 `文件:行号`
- ✅ 必须 grep / 读 代码·配置·schema 真值，每条结论带来源
- ✅ 读不到真值时，必须直说"我没读 X，本结论不可信"，**不允许编**

**自检红线**：脑子里冒出"我记得""应该是"，或想引用旧文档 → 立即停，去验证。写完一段没带来源 → 删掉或补来源。

这条要做成**绝对**的，不能软化。一旦留"通常情况下"的口子，漂移就从那个口子钻进来。

### 第三原则·推论:摘要永不作为代码真值

历史决策记录、活交接层、记忆索引（见第一原则）是允许的——它们记录的是"何时做了什么决定"，属于史实层。

但任何摘要、跨 session 自动压缩、向量化检索结果，**永不能作为当前代码事实的来源**。涉及代码事实的任何结论，必须当下直接从代码读出，不接受从摘要继承。

这是真值优先在"记忆"维度的直接推论：摘要是②号（认知层）漂移的天然喂养机制——可作历史记录，不可作真值源。

**操作落法 A — compaction / session 续接后必须 re-verify。** 如本 session 是续接 / 曾被 compact,在任何动手之前,**所有从摘要继承的指称必须当下重新从源头验证** —— unit 名跑 `systemctl list-units`、文件路径跑 `ls`、版本号读 `config.json`、服务 ID 读运行状态,**每一条都要源**,无论它读起来多熟悉。熟悉 ≠ 真值。

**操作落法 B — ACGM 仪式合规的真实判据。** 任何 ACGM 仪式动作(grounding 报告 / fact statement / 5 步声明)的合规判定,在于**仪式里每条具体技术事实是否当下取证**,而不是仪式格式是否齐全。"格式合规"不等于"实质合规";只有每条事实在本 session 同一段对话内有对应工具调用(grep / cat / systemctl list-units / ls 等)在前,且该调用的输出**包含被引用的事实**,才算实质合规。

### 第三原则·推论:操作真值

> **建议性条款 · 单次直接事件支撑** —— 按 CONTRIBUTING 2-独立证据规则,过渡期 ≤30 天;待第二独立事件出现后转主条款,或降级到元观察。

文档里写"使用 X"必须带 `文件:行号`;但命令里写 `systemctl stop X` 长期没有同等约束。这是真值优先在**操作平面**的盲区。

命令里的指称(unit 名 / 文件路径 / 服务 ID / PID / 容器 / 主机)和文档里的措辞享**同一证据标准**。任何破坏性 / 状态变更操作之前,命令里每一个标识符的来源必须**当下可指** —— 指向本 session 内某次 `systemctl list-units` 输出的第 N 行、或 `ls` 列表、或 `cat config` 某字段 —— **不许凭 session 内别名或印象复用**。

> **失败模式:** agent 把一次性建立的别名("27B" / "主模型" / "缓存目录")当成真实标识符,直接发命令。真实名 10 秒内可拿到,但不查 = ② 在操作平面。

### 第三原则·推论:async / 后置自我监控

> **建议性条款 · 单次直接事件支撑** —— 同 2-独立证据规则。

truth-first 关心结论的来源,grounding 关心动手前的状态,**但都没说动手之后呢**。

你启动的任何**后台进程 / 长跑任务 / 状态变更操作**,它的完成或失败验证**由你拥有**,直到该状态被显式验证。

- "已启动" ≠ "在跑":`systemctl start X` 之后必查 `is-active` + 资源占用,**不只看 exit code**。
- "返回 0" ≠ "达到意图":destructive 操作之后必查是否命中预期对象、是否产生预期效果。
- 后台任务必登记:**何时回头查、用什么命令查存活/进度**;若涉及自启动循环,必须显式声明监控间隔。

**三层结构定位:** 本条是规范层条款。机制层(PreToolUse hook)能做的是"承诺留痕" —— agent 当场声明"执行后我会跑 X 来验证";**几小时后是否真去查,由审计层关闭**。这种"机制部分覆盖、规范完整宣称"的形态,正是元立场所述的样板。

## 6. 第四原则：session 启动 grounding 仪式（先验证再动手）

每个 session（新开 / 续接 / 接半截活）开工前强制走：

1. **读宪法 + 根规则文件**（必读，不跳读）
2. **判断本次落在哪个轨道/范围**（见第八原则），加读对应层文档
3. **报告 5 件事，然后等人确认再动手**：
   - 我落在哪个轨道
   - `git log` + `git status` 显示的现状
   - 我从实际读代码看到的相关结构（不凭印象）
   - 我打算改的文件清单（具体路径）
   - 我打算的执行步骤
4. 改完跑验证脚本
5. 收尾报告 + commit 草稿，等人审批再 commit

把这套写成一份**通用启动话术**，固化成文件。开任何 session 第一句贴它，末尾附具体任务。这样你不用每次重新解释规则，新 session 自动接入治理。**这一步能砍掉一多半的交接成本。**

> 关键：**先转述，再动手。** 转述阶段暴露的偏差，比写完代码再发现便宜一个数量级。

### 第四原则·推论:破坏性操作前必先 grounding

> **建议性条款 · 单次直接事件支撑** —— 同 2-独立证据规则。

5 步 grounding 不只在 session 启动时跑;在**每次破坏性 / 状态变更操作之前**也必须跑,至少做 5 步里的第 1-3 步:

1. **读相关治理规则:** 本操作类的红线(如真值优先的操作真值推论、本仓的范围边界)。
2. **识别工作树 / 分支 / 轨道:** 本操作所在的工作树和当前分支(对应**第七原则·推论:工作树纪律**)、落在哪个轨道、影响哪个范围。**若在主工作树且非主干 = 第七原则·推论 异常,立刻停下,不得执行此破坏性操作。**
3. **报告当前真实状态:** `is-active` / GPU 占用 / 现有进程 / 文件 mtime —— **实读,不凭印象**。

"破坏性 / 状态变更"包括但不限于:`systemctl stop/start/disable/restart`、`rm -rf`、`git push --force` / `git reset --hard`、`drop/truncate/delete from`、`dd`、`mkfs`、`kill -9`、模型加载/卸载、服务启停。

> 与 §第七原则·推论 的呼应:工作树纪律说"主树永远=主干";本条让这条主张在**每次破坏性操作之前**都被重新验证,而不只是 session 启动时验一次。

## 7. 第五原则：不过度执行（暴露歧义，不蛮干）

AI 在指令有歧义时倾向"猜一个最省事的解释然后干完"。这会把一个含糊的"删掉它"执行成"摧毁了用户没评估过的内容"。

规则：
- 指令涉及**不可逆操作**（删文件/删分支/删历史）前，若有任何歧义或基于未验证假设，**先验证，验证不过就停下问**。
- 发现某改动的影响面比表面大（牵连多文档、含被追踪的决策），**列出影响面让人定**，不要替人做判断。
- 销毁性操作要有**硬人工检查点**：列清单 + 写回滚 + 引用授权原话。绕过安全检查点本身就是①号漂移。
- 区分"用户的业务判断"和"AI 的执行"：前者（要不要做某事、某内容有没有价值）是人的；AI 只负责把已定的事做对。

> 真实收益：一个会"停下来确认"的 AI，比一个"勇往直前"的 AI，少返工得多。返工是最贵的 token。

---

## 8. 第六原则：按"轨道"隔离工作

不同性质的工作有不同的认知上下文、验证方法、风险特征、回滚机制。混在一个 session 里改，决策互相干扰，事后归因困难。

按你项目的实际，把工作分成若干**轨道**。每个轨道：
- 有独立的工作目录范围
- 有独立的验证方法（代码靠测试/构建；内容/AI行为靠评测集 eval/baseline）
- 有独立的风险特征（代码 bug 是硬故障可定位；质量回归是软故障难发现）
- 一个 session 只在一个轨道工作；跨轨道任务拆成连续多个 session

> 例（仅示意，按你项目改）：典型项目至少分"代码轨道"和"内容/AI行为轨道"。如果产品核心价值是 AI 行为本身（prompt/知识/检索），那条轨道必须有**评测基线**——否则每次调 prompt 都是凭感觉，质量回归发现得晚、损失大。**在 AI 行为成为主战场之前，先建评测基础设施。**

物理隔离手段（按需，从轻到重）：
- 每个轨道一份子目录规则文件（薄指针 + 该目录专属规则），进哪个目录自动落哪个轨道
- 工作树（worktree）不是"高并发才用"的可选项，而是第七原则"一个主干永不腐化"的强制机制（见"第七原则·推论:工作树纪律"）；**单用户并行同样必须用**
- 工作树之间靠契约文件协同，不靠口头交接

## 9. 第七原则：一个主干，永不腐化

- **主干唯一**（通常是 `main`），是项目当前真值的规范来源。
- **禁止让主干腐化**：被某个功能分支事实上取代而主干停滞 = 违规。
- **治理/通用真值文件只在主干 author**。功能分支只消费；要改治理 → 提案 → 在主干落地 → 功能分支同步。在功能分支上 author 治理 = 结构放置漂移的根本错误。
- 功能分支从主干 fork，隔离开发，完成 merge 回主干。
- 任何 session 启动若发现自己落在腐化主干 / 治理落后于某分支 → 立即停下报告，不"将就着干"。

> 这条是最反直觉、也最容易被违反的。治理体系第一次往往就是建在一个功能分支上的——因为开始建治理那一刻，恰好在做某个功能。务必尽早把它归位到主干。

### 第七原则·推论:工作树纪律 —— 让"一个主干永不腐化"真正可执行

第七原则说"治理只在主干 author、主干不腐化"，但**没说靠什么物理保证**。答案是工作树纪律。

核心认知（反直觉、最易忽视）：**"当前在哪个分支"是工作树（那个目录）的属性，不是 session/agent 的属性。** 因此——即使只有你一个人、没有多人并发——只要两条线共用一个工作目录（来回 `checkout`，或两个 agent session 指向同一目录），切分支就会把对方脚下的文件换掉，治理文档会"凭空"落到错误分支。这不是高并发才有的问题，**单用户并行就会触发**。

纪律：
- **一条长期工作线 = 一个独立工作树目录，钉死自己的分支**；该目录内不切别的分支。
- **主工作树永远 = 主干**，是治理/通用真值的唯一 author 居所（对应第七原则）与版本汇聚/打 tag 点；**严禁在主树切到别的分支**。
- 平台线、功能线各自独立工作树；新线从主干 fork 出分支 + 新工作树。
- **汇聚只经主干**：功能成熟 merge 回主干；各线定期 merge 主干以接收治理与功能更新；**禁止线间直接 merge**（拓扑是绕主干的星形，不是网状）。
- **不在 git 内的基建（部署/外部服务/环境）不靠分支版本化**，而是在主干上以 ADR + 快照记录。一个"功能"常分两半：代码半（在 git，功能工作树）+ 基建半（在 git 外，在主干治理文档里记），二者不可混为一谈。
- session 启动 grounding 必须确认：自己在正确的工作树、正确的分支；**主树若不在主干 = 异常，立即停下报告**（把第七原则"落在腐化主干就停"延伸到工作树放置层）。

物理隔离 + 主干汇聚，正好对应真实的多线开发节奏：多条线长期并行，到版本节点经主干汇聚，之后再并行、再汇聚，循环往复。

> 工作树纪律是第七原则的操作落法，不替代它——第七原则管"治理住哪"，本推论管"用什么物理手段保证它真住那"。

## 10. 第八原则：范围边界

明确什么内容配进这个开发仓库。

- **判据一句话**：这件事是为了**软件能开发/上线/运行**，还是为了**别的**（公司怎么经营、商业战略、与软件无关的规划）？前者 IN，后者 OUT。
- 技术基础设施（域名/邮件/部署/合规备案等直接关系软件能否上线运行的）虽含"非代码"成分，但服务于软件 → IN。
- 纯经营/战略/组织决策 → OUT，不在本仓库 author 或追踪；已混入的，经确认后删除或移出。
- AI 不主动在开发仓库产出本不属于这里的内容。
- 边界有疑义 → 按判据；仍不清 → 停下问，**不擅自扩大删除**。

> **关于这条判据本身（给采用者）：** "软件 IN / 其他 OUT" 是本方法论的**默认主张**，也是保住本仓自身干净的那条线。但范围边界天然因项目而异——若你的项目需要不同的 IN/OUT 判据，这一条允许你按自己情况重写（见 §14：范围边界属"按你项目实例化"项）。重写它不算漂移；**没有显式边界**才算。

## 元观察 / Meta-observations

四类漂移(①②③④)回答"**什么**失效了"——失效形态分类。下面的元观察回答"失效**怎么**发生"与"治理**为什么**会随时间失效"——与失效分类**正交**,属于方法论自身的认知与设计观察。它们不是漂移分类的扩展,故单列章节。

随时间增长 —— 发现新的方法论本身层面的观察,加进来。

### 元观察 1:假装合规(performative compliance)

Agent 可以产生**合规的外观**而无**合规的实质** —— 贴出完整的 grounding 报告 / fact statement / 仪式声明,但其中具体技术事实未当下取证,仅从摘要或印象继承。这是 ② 的一种成熟特化形态,需单独警觉。

判定合规必须基于**仪式里每条事实的当下取证**,而非仪式格式齐全(详见第三原则·推论:摘要不作真值 · 操作落法 B)。

**识别判据(可机械检查):** fact statement / grounding 报告里每一个被引用的具体事实(数字、unit 名、版本号、状态),必须在**本 session 同一段对话内**有对应工具调用(grep / cat / systemctl list-units / ls 等)在前,且该工具调用的输出**包含被引用的事实**。否则即使写了 `文件:行号` 字段,仍属于形式合规。

**当前证据状态:** 预测,**强造假形态(声称跑了 grep 但实际没跑)未观察到**。已观察到的弱形态(摘要继承未 re-verify)由"摘要不作真值"覆盖。强造假一旦观察到,按 2-独立证据规则升级为 §3 独立推论。

### 元观察 2:注入饱和(injection saturation)

Hook / SessionStart 文本 / PreToolUse 提问注入越频繁,agent 越快学会**语义忽略** —— 机制设计必须节制,精准命中而非碾压式覆盖。这是 ACGM 自身可持续性的约束条件。

**设计后果:**

- 每个新 hook 的引入必须有明确的失效证据支撑,**不许"以防万一"式增加**。
- 同一种提醒不许在不同 hook 之间重复触发(SessionStart 与 PreToolUse 不许说同一句话)。
- **白名单类 hook 优先于全量 hook**(PreToolUse 只盯破坏性 Bash,而不是所有 Bash)。
- 单 session 内同一 hook 的触发频率高于 5 次以上 = 设计气味,需要审查。

**当前证据状态:** 设计预防原则,**预测但未观察到具体注入饱和事件**。本方法论自身的 hook 设计已据此约束(PreToolUse 仅盯破坏性 Bash;SessionStart 仅在 session 启动/续接时触发)。

## 11. 权威与修订模型

- **宪法只有人能改。** AI 试图改宪法 = 违规。新增条款追加、注明日期与触发案例，不挪动既有编号；推翻条款要在修订记录里说明原因。
- **决策走 ADR。** 一决策一文件，append-only。
- **被取代 vs 被废除要分清**：
  - *Superseded*（被取代）：决策仍有效，只是被更新的决策替换 → 标 `Superseded by ADR-NNNN`
  - *Withdrawn*（被废除）：决策本身作废，不是被替换 → 标 `Withdrawn`
- 跨切面契约改动：必须单独 commit + 写影响面 + 跑验证 + 人审批后才能合并。
- 重要的边界/范围裁定也要留可追溯记录（追加进对应 ADR 即可）。

## 12. 落地配方（在新项目上 bootstrap）

按这个顺序建，从最小可用到完整：

**第 0 步 · 先审计（只读，不写代码）**
开一个只做审计的 session：通读现有说明文件 + 历史交接 + 对照当前代码。输出三份清单：① 还成立的 ② 已过时该删的 ③ 矛盾需人裁定的。这份产出是后续重构的输入。

**第 1 步 · 建宪法**
把"还成立"里不变的部分（红线、角色边界、范围边界、四类漂移防线）写进宪法。宪法只有人能改。

**第 2 步 · 重写根规则文件**
砍掉所有事实，只留：元规则 + 指针 + 行为约束。控制在一页左右。

**第 3 步 · 建决策日志**
建 ADR 目录 + 模板 + 索引。把已发生的关键决策补录成 ADR（事后补录要标"基于当时已知信息重建"，并尤其警惕②号漂移）。被推翻的标清 Superseded/Withdrawn。

**第 4 步 · 建快照机制**
当前状态文档**由代码扫描生成**，顶部标"基于 commit X"。配套写几个**漂移检查脚本**（哪怕只是 grep 验证文档写的和代码一致）——脚本和文档一起建，不是事后补。

**第 5 步 · 建启动话术文件**
把第六原则的 grounding 仪式固化成一份通用启动话术 + 几个典型场景模板。以后开 session 直接复制。

**第 6 步 · 划轨道 + 范围边界**
按项目实际分轨道，每轨道一份薄子目录规则文件。明确 IN/OUT。

**第 7 步 · 定期审计成习惯**
每个版本封存时重跑快照；每隔几个版本跑一次根规则文件审计。把"漂移显性化"做成周期机制。

> 在已经积累了相当规模的项目上，花 1-2 天做这次重构是高回报的——后续每个版本省下的重复对账和 token 都很可观。

## 13. 失败模式 & 自检清单

这套体系本身也会被违反。常见失败：

- 把事实又写回了根规则文件 → 第 4 原则失守
- 凭印象写技术结论没给来源 → 第 5 原则失守（最高频）
- session 不走 grounding 直接动手 → 第 6 原则失守
- 把含糊指令蛮干执行成不可逆破坏 → 第 7 原则失守
- 治理又长在功能分支、主干腐化 → 第 9 原则失守
- 跨切面契约偷偷改不声明 → 第 11 失守

**给审查者的红线**：看到任何"我认为/应该/通常/我记得"措辞、任何技术结论无来源、任何被推翻文档被当真值引用——直接拒绝，不讨论。

## 14. 哪些拿去直接用，哪些要按你项目改

**直接拿去用（通用）**：
- 第 1 节问题诊断、第 2 节四类漂移分类
- 第 3-7 原则（分层 / 根文件只装规则 / 真值优先 / grounding 仪式 / 不过度执行）
- 第 9 原则（一个主干不腐化）、第 11 权威模型、第 12 落地配方、第 13 自检

**按你项目实例化（特质化）**：
- 第 8 原则的**轨道怎么分**（取决于你项目的核心价值在哪、有几个认知上下文）
- 第 10 范围边界的 IN/OUT 具体清单
- 跨切面契约具体是哪些协议（多端？多服务？API？数据格式？）
- 各层文档的具体目录命名、ADR 编号规则、验证脚本写什么
- 红线具体内容（这是你的产品/合规决定的）

> 原则是骨架，可迁移；血肉是你项目自己的。**别把别人的轨道划分/契约清单照抄**——那是别人项目的特质，照抄就是另一种漂移。

## 15. 给持怀疑态度的同事一段话

> "这套东西看着很重，真有必要？"
>
> 到一定规模（多版本、多 session、可能多人/多分支），AI 辅助开发的腐化不是会不会，是迟早。腐化一旦发生，你花在交接对账、信息核对、返工上的时间会指数增长，项目推进效率反而越来越低。
>
> 一个减少 30% 重复工作和返工的工作流，等于免费给你一个更快更便宜的模型。前期 1-2 天的结构投入，换的是后面 6-12 个月不再被同一类问题反复拖住。
>
> 这套方法论投资回报率，比纠结选哪个模型高得多。

## 附:本方法论的来源与边界

- 本文是从一个真实长周期项目的治理实践中提炼的**逻辑骨架**，已剥离该项目所有业务特质。
- 它不是理论，是踩坑沉淀——包括"建治理体系时自己违反治理"这种元级翻车。
- **关于成本(未实测,作者判断)**:这套纪律很可能让单位任务消耗*更多* token——更多读代码、验证、转述、停下确认;但它换来更少的错误和走错路,从而压缩最贵的那部分——返工。作者本人不受 token 约束、未做实测,具体增减请在你自己的项目上体验判断。
- 你的项目特质不同，**实例化时务必重新设计轨道与契约，不要照抄具体清单**。
- 这份文件本身也该被它自己的规则管：它是方法论（规则层），不放具体事实；要改先想清楚改的是"通用逻辑"还是"某项目特质"——后者不该进这份文件。
