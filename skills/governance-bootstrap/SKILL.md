# Governance Bootstrap

> **This is a checklist and a set of phrasings — not an autonomous run.** Every step
> below is a human decision and a human approval. The AI proposes; the human rules.
> Nothing irreversible happens without explicit authorization (`truth-first` applies
> throughout). Expect this to span multiple sessions, not one.

Build from minimum-viable to complete, in order:

0. **Audit first (read-only, no code).** One audit-only session: read existing rules
   files + history/handoffs, compare against current code. Output 3 lists — ① still
   valid ② outdated, delete ③ contradictory, needs human ruling. This is the input
   to everything after.
1. **Build the constitution.** Write the unchanging parts of "still valid" (redlines,
   role boundaries, scope boundary, the four-drift defenses) into it. Only humans
   change the constitution. → `templates/CONSTITUTION.skeleton.md`
2. **Rewrite the root rules file.** Cut every fact. Keep only meta-rules + pointers +
   behavior constraints. About one page. → `templates/SESSION_START.skeleton.md` feeds this.
3. **Build the decision log.** ADR dir + template + index. Backfill key decisions as
   ADRs (mark "reconstructed from then-known info"; beware drift #2). Mark
   Superseded/Withdrawn. → `templates/ADR._TEMPLATE.md`
4. **Build the snapshot mechanism.** Current-state docs are *generated by code scan*,
   top-stamped "based on commit X". Write drift-check scripts alongside the docs, not
   after. → `templates/drift-check.stub.js`
5. **Build the startup-phrase file.** Solidify the grounding ritual into a generic
   startup phrase + scenario templates. (See the `session-grounding` skill.)
6. **Draw tracks + scope boundary.** Split tracks per *your* project's reality; one
   thin sub-dir rules file per track. Define IN/OUT explicitly.
7. **Make periodic audit a habit.** Re-run the snapshot at each version freeze;
   re-run the root-file audit every few versions.

> The principles are the skeleton (portable). The tracks and contracts are your
> project's flesh — design them fresh, do not copy someone else's.

# 治理 bootstrap

> **这是 checklist 和话术,不是自主执行。** 下面每一步都是人的决策、人的审批。AI 出
> 提案,人来裁定。任何不可逆动作没有明确授权不发生(全程适用 `truth-first`)。预期
> 跨多个 session,不是一次干完。

按从最小可用到完整的顺序:

0. **先审计(只读,不写代码)。** 一个只做审计的 session:通读现有规则文件+历史交接,
   对照当前代码。输出三份清单——① 还成立 ② 已过时该删 ③ 矛盾需人裁定。后续重构的输入。
1. **建宪法。** 把"还成立"里不变的部分(红线、角色边界、范围边界、四类漂移防线)写进
   去。宪法只有人能改。→ `templates/CONSTITUTION.skeleton.md`
2. **重写根规则文件。** 砍掉所有事实,只留 元规则 + 指针 + 行为约束。约一页。
3. **建决策日志。** ADR 目录+模板+索引。关键决策补录为 ADR(标"基于当时已知信息重建",
   警惕②号漂移),标清 Superseded/Withdrawn。→ `templates/ADR._TEMPLATE.md`
4. **建快照机制。** 现状文档**由代码扫描生成**,顶部标"基于 commit X"。漂移检查脚本和
   文档一起建,不是事后补。→ `templates/drift-check.stub.js`
5. **建启动话术文件。** 把 grounding 仪式固化成通用启动话术+场景模板。(见
   `session-grounding` skill。)
6. **划轨道 + 范围边界。** 按**你**项目实际分轨道,每轨道一份薄子目录规则文件。明确
   IN/OUT。
7. **定期审计成习惯。** 每次版本封存重跑快照;每隔几个版本跑一次根文件审计。

> 原则是骨架(可迁移);轨道与契约是你项目的血肉——重新设计,别照抄别人的。
